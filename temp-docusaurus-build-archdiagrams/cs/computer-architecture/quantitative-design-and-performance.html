Quantitative Design and Performance

Computer architecture is not mainly a catalog of clever hardware structures. In the Hennessy and Patterson style, it is an engineering discipline built around measurement, trade-offs, and explicit assumptions. A proposed pipeline, cache, instruction set, or multiprocessor mechanism is only meaningful after asking: what workload does it improve, by how much, at what cost, and under which power or reliability limits?

This page introduces the quantitative habits used throughout the rest of the architecture notes. The same ideas reappear in cache design, branch prediction, vector machines, warehouse-scale computers, and storage systems: define the metric, decompose execution time, estimate the affected fraction, and distrust improvements that optimize a narrow component while hurting the complete system.

Definitions​

The most important performance metric is execution time. If machine XXX finishes a fixed program in less time than machine YYY, then XXX is faster for that program. Performance is the reciprocal of execution time:

PerformanceX=1Execution timeX\mathrm{Performance}_X = \frac{1}{\mathrm{Execution\ time}_X}PerformanceX​=Execution timeX​1​

The speedup of a new design over an old design is:

Speedup=Execution timeoldExecution timenew=PerformancenewPerformanceold\mathrm{Speedup} = \frac{\mathrm{Execution\ time}_{old}}{\mathrm{Execution\ time}_{new}} = \frac{\mathrm{Performance}_{new}}{\mathrm{Performance}_{old}}Speedup=Execution timenew​Execution timeold​​=Performanceold​Performancenew​​

CPU execution time is commonly decomposed as:

CPU time=Instruction count×CPI×Clock cycle time\mathrm{CPU\ time} = \mathrm{Instruction\ count} \times \mathrm{CPI} \times \mathrm{Clock\ cycle\ time}CPU time=Instruction count×CPI×Clock cycle time

Equivalently, because Clock rate=1/Clock cycle time\mathrm{Clock\ rate}=1/\mathrm{Clock\ cycle\ time}Clock rate=1/Clock cycle time:

CPU time=Instruction count×CPIClock rate\mathrm{CPU\ time} = \frac{\mathrm{Instruction\ count} \times \mathrm{CPI}}{\mathrm{Clock\ rate}}CPU time=Clock rateInstruction count×CPI​

For an instruction mix with classes iii, total CPI is a weighted average:

CPI=∑iICi×CPIi∑iICi\mathrm{CPI} = \frac{\sum_i IC_i \times CPI_i}{\sum_i IC_i}CPI=∑i​ICi​∑i​ICi​×CPIi​​

where ICiIC_iICi​ is the number of executed instructions of class iii. This formula is useful because it makes a hidden workload assumption explicit: a faster floating-point unit helps only in proportion to how much floating-point work is actually executed.

Benchmarks are controlled workloads used to compare systems. A good benchmark suite includes real programs, reproducible build and run conditions, input sets large enough to exercise the system, and reporting rules that prevent benchmark-specific tricks from replacing genuine design improvement. Throughput benchmarks measure completed jobs per unit time; latency benchmarks measure time for one job.

The geometric mean is often used to summarize normalized benchmark ratios:

Geomean(r1,…,rn)=(∏k=1nrk)1/n\mathrm{Geomean}(r_1,\ldots,r_n) = \left(\prod_{k=1}^{n} r_k\right)^{1/n}Geomean(r1​,…,rn​)=(k=1∏n​rk​)1/n

It is preferred for ratios because it treats reciprocal changes symmetrically: a 2×2\times2× improvement and a 2×2\times2× slowdown average to 1×1\times1×, not to an arithmetic mean of 1.25×1.25\times1.25×.

Key results​

Amdahl's law is the central warning against over-valuing local optimization. If fraction fff of the original execution time can be improved by factor sss, the overall speedup is:

Speedupoverall=1(1−f)+fs\mathrm{Speedup}_{overall} = \frac{1}{(1-f)+\frac{f}{s}}Speedupoverall​=(1−f)+sf​1​

The proof is a direct normalization. Let old execution time be 111. The unaffected part remains (1−f)(1-f)(1−f). The improved part becomes f/sf/sf/s. The new time is (1−f)+f/s(1-f)+f/s(1−f)+f/s, so speedup is the reciprocal.

The same idea applies to parallel processors. If fraction ppp can run perfectly on NNN processors and the rest is serial, then:

Speedup(N)=1(1−p)+pN\mathrm{Speedup}(N)= \frac{1}{(1-p)+\frac{p}{N}}Speedup(N)=(1−p)+Np​1​

As N→∞N \to \inftyN→∞, the limiting speedup is 1/(1−p)1/(1-p)1/(1−p). A program that is 95%95\%95% parallel has an upper bound of 20×20\times20× no matter how many processors are purchased, unless the serial fraction is reduced.

The CPU time equation also explains why comparing only clock rates is a pitfall. A design with a higher clock rate can still lose if it executes more instructions or has a worse CPI. A compressed instruction set may reduce instruction count and instruction-cache misses but complicate decoding. A deeper pipeline may shorten clock cycle time but increase branch misprediction penalties. A wider issue processor may reduce CPI on one benchmark and waste power on another.

A useful performance argument therefore has three parts:

  • Workload: the exact program or benchmark family being optimized.
  • Decomposition: the pieces of time, CPI, miss penalty, or stall time that explain the result.
  • Constraint: cost, energy, die area, complexity, verification risk, and availability.

The decomposition step is also how architects avoid double counting. Suppose a branch predictor reduces branch stalls, and a larger instruction window also reduces the visible cost of some branch stalls by finding independent work. Adding the two isolated speedups will overstate the combined result, because both features attack some of the same baseline cycles. A safer method is to build a single baseline time budget, subtract or scale each component once, and then recompute total time.

Another useful habit is to distinguish latency-oriented and throughput-oriented claims. A faster core may reduce the response time of one request, while more cores may increase requests per second. These are both performance improvements, but they answer different questions. In a web service, the better design may be the one that delivers more throughput at a fixed 99th percentile latency. In an interactive application, the better design may be the one that lowers one user's response time even if peak throughput is unchanged.

Finally, measurements should be reproducible. A benchmark report should describe hardware, compiler version, flags, memory size, operating system, input set, thermal settings, and whether turbo or automatic parallelization was enabled. Without those details, a number may be true for one run but not useful for engineering comparison.

Worked example 1: CPI from an instruction mix​

Problem: A program executes 2.0 billion instructions on a 2.5 GHz processor. The instruction mix is 45% integer ALU at CPI 1, 25% loads/stores at CPI 2, 20% branches at CPI 1.5, and 10% floating-point at CPI 4. Find average CPI and CPU time.

Method:

  1. Convert percentages into weights.
wint=0.45wmem=0.25wbr=0.20wfp=0.10\begin{aligned} w_{int} &= 0.45 \\ w_{mem} &= 0.25 \\ w_{br} &= 0.20 \\ w_{fp} &= 0.10 \end{aligned}wint​wmem​wbr​wfp​​=0.45=0.25=0.20=0.10​
  1. Compute weighted CPI.
CPI=0.45(1)+0.25(2)+0.20(1.5)+0.10(4)=0.45+0.50+0.30+0.40=1.65\begin{aligned} \mathrm{CPI} &= 0.45(1) + 0.25(2) + 0.20(1.5) + 0.10(4) \\ &= 0.45 + 0.50 + 0.30 + 0.40 \\ &= 1.65 \end{aligned}CPI​=0.45(1)+0.25(2)+0.20(1.5)+0.10(4)=0.45+0.50+0.30+0.40=1.65​
  1. Compute total cycles.
Cycles=IC×CPI=2.0×109×1.65=3.30×109\begin{aligned} \mathrm{Cycles} &= IC \times CPI \\ &= 2.0 \times 10^9 \times 1.65 \\ &= 3.30 \times 10^9 \end{aligned}Cycles​=IC×CPI=2.0×109×1.65=3.30×109​
  1. Divide by clock rate.
CPU time=3.30×1092.5×109=1.32 s\begin{aligned} \mathrm{CPU\ time} &= \frac{3.30 \times 10^9}{2.5 \times 10^9} \\ &= 1.32\ \mathrm{s} \end{aligned}CPU time​=2.5×1093.30×109​=1.32 s​

Checked answer: The average CPI is 1.651.651.65, and the CPU time is 1.321.321.32 seconds. A quick sanity check says the time should be slightly more than one second because the program needs 3.3 billion cycles and the processor supplies 2.5 billion cycles per second.

Worked example 2: Amdahl's law with a faster floating-point unit​

Problem: In a baseline run, floating-point operations account for 30% of execution time. A new floating-point unit makes that part 5×5\times5× faster, but it does not change the rest of the program. What is the overall speedup? What fraction of the new execution time is floating-point?

Method:

  1. Normalize old execution time to 111.
Told=1.00f=0.30s=5\begin{aligned} T_{old} &= 1.00 \\ f &= 0.30 \\ s &= 5 \end{aligned}Told​fs​=1.00=0.30=5​
  1. Compute new time.
Tnew=(1−f)+fs=0.70+0.305=0.70+0.06=0.76\begin{aligned} T_{new} &= (1-f)+\frac{f}{s} \\ &= 0.70 + \frac{0.30}{5} \\ &= 0.70 + 0.06 \\ &= 0.76 \end{aligned}Tnew​​=(1−f)+sf​=0.70+50.30​=0.70+0.06=0.76​
  1. Compute speedup.
Speedup=10.76=1.3158\mathrm{Speedup}=\frac{1}{0.76}=1.3158Speedup=0.761​=1.3158
  1. Compute the new floating-point fraction.
FP fractionnew=0.30/50.76=0.060.76=0.0789\begin{aligned} \mathrm{FP\ fraction}_{new} &= \frac{0.30/5}{0.76} \\ &= \frac{0.06}{0.76} \\ &= 0.0789 \end{aligned}FP fractionnew​​=0.760.30/5​=0.760.06​=0.0789​

Checked answer: The overall speedup is about 1.32×1.32\times1.32×, not 5×5\times5×. After the change, floating-point work is only about 7.9%7.9\%7.9% of execution time, so further floating-point acceleration would have much less value.

Code​

from math import prod

def weighted_cpi(classes):
    total_ic = sum(ic for ic, _ in classes)
    total_cycles = sum(ic * cpi for ic, cpi in classes)
    return total_cycles / total_ic

def amdahl(fraction, local_speedup):
    return 1.0 / ((1.0 - fraction) + fraction / local_speedup)

def geomean(ratios):
    return prod(ratios) ** (1.0 / len(ratios))

mix = [
    (0.45 * 2_000_000_000, 1.0),
    (0.25 * 2_000_000_000, 2.0),
    (0.20 * 2_000_000_000, 1.5),
    (0.10 * 2_000_000_000, 4.0),
]

cpi = weighted_cpi(mix)
cpu_time = 2_000_000_000 * cpi / 2_500_000_000
print(f"CPI = {cpi:.2f}, CPU time = {cpu_time:.2f} s")
print(f"Amdahl speedup = {amdahl(0.30, 5):.3f}x")
print(f"Geomean = {geomean([1.2, 0.9, 1.5, 1.1]):.3f}x")

Common pitfalls​

  • Optimizing a component and reporting the component speedup as if it were system speedup.
  • Comparing clock rates without comparing instruction count and CPI.
  • Using arithmetic means for normalized benchmark ratios.
  • Forgetting that benchmark inputs can change cache behavior and branch behavior.
  • Treating peak throughput as useful throughput when response-time constraints are violated.
  • Ignoring the energy, cost, and verification cost of a performance feature.
